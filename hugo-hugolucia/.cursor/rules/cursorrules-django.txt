# 5) Multi-tenant + RLS rules (fixed)
- Every business table MUST have organisation_id UUID NOT NULL (unless explicitly global like rare static enum tables; avoid globals in this POC).
- RLS is enforced in PostgreSQL, not simulated.
- At the start of each request after authentication, set tenant context per DB transaction:
  - execute: "SET LOCAL app.organisation_id = %s"
- Implement this via Django middleware + DB cursor execution, ensuring it runs:
  - AFTER authentication
  - BEFORE any ORM queries in the request
- All RLS policies must use:
  - organisation_id = current_setting('app.organisation_id', true)::uuid
  Using the (.., true) form to avoid exceptions when unset (returns NULL).
- Add cross-tenant tests early (two orgs A/B). Any A->B access must be rejected.
- App-level checks remain mandatory (defence in depth), especially on write paths.

# 6) Privacy & sharing (fixed)
- Learner can see all their verbatim, traces and evidence.
- Tutor/trainer/coordinator can ONLY access what the learner shared explicitly:
  - share_summary, share_evidence, share_verbatim flags
- Defaults: share_summary=false, share_evidence=false, share_verbatim=false.
- Implement authorization checks in Django services (domain layer) and DRF permissions, not only in UI.
- No raw hugo_message.content in logs; only metadata and status.

# 7) Hugo + RAG rules (fixed)
- Hugo must ask exactly 1 question per message (guardrail).
- Messages should be concise (no essay-style answers).
- RAG is question-driven: retrieved chunks are used to craft situated follow-up questions, not to teach full lessons.
- Only class library ACTIVE documents are searchable.
- Prefer procedure/checklist/securite/qualite tags when risk/compliance intent is detected.
- Store rag_citation for each Hugo message that uses RAG chunks.
