# Postgres RLS & Tenant Context

## Goal
Guarantee hard isolation between organisations (OF) in the same database, as required by SPEC_POC_v1.5.

## Tenant context mechanism (mandatory)
For every request after authentication, for every DB transaction:
- Execute: `SET LOCAL app.organisation_id = '<org_uuid>'`

Important:
- `SET LOCAL` is scoped to the current transaction.
- Ensure you set it for each transaction or session lifecycle.

## RLS policy pattern
Use `current_setting` with the safe form:
- `current_setting('app.organisation_id', true)`

Reason:
- Without the second argument, `current_setting()` can throw if parameter is not set.
- With `true`, it returns NULL instead.

### Example policy snippet
For a table with `organisation_id`:

```sql
ALTER TABLE my_table ENABLE ROW LEVEL SECURITY;

CREATE POLICY my_table_tenant_isolation
ON my_table
USING (organisation_id = current_setting('app.organisation_id', true)::uuid)
WITH CHECK (organisation_id = current_setting('app.organisation_id', true)::uuid);
```

## Which tables must have RLS
All business tables (POC scope):
- organisation, user, audit_log
- group, group_membership, tutor_learner_link
- hugo_session, hugo_message
- trace, evidence
- document, group_document, document_chunk, rag_citation
- referential, referential_item, scale, scale_level, referential_config, referential_item_overlay
- learner_state
- export_template, export_run

POST-POC (to be RLS-protected when added):
- qualiopi_indicator_mapping, qualiopi_evidence_bundle
- referential_mapping, export_template (extended with column_mapping), external_identity
- schedule_proposal, schedule_event, resource_booking, schedule_change_set, schedule_event_log
- webhook_subscription, webhook_delivery (if introduced)

## Cross-tenant tests (required)
Test scenario:
- Create org A + org B.
- Create one object in each (e.g., group, trace).
- Authenticate as A user.
- Verify:
  - cannot read B objects
  - cannot create objects with organisation_id=B
- Repeat for sensitive endpoints (traces, evidence, documents, exports, audit_log).

## Performance note
RLS adds overhead; keep organisation_id indexed.
Keep policies simple and add indexes if needed.

# RAG Configuration (Class Library, Question-Driven)

## Scope
- RAG uses ONLY ACTIVE documents attached to a class (group library).
- RAG supports:
  - long course materials (majority)
  - critical procedures/checklists/sécurité (anti-absurd tests)

## Document tagging (required)
Allowed tags (normalize to lowercase):
- cours
- procedure
- checklist
- securite
- qualite
- reglementaire

## Extraction (best effort)
Strongly supported (POC guarantee):
- PDF text (selectable text), extracted via pypdf or equivalent.

Scanned PDFs without OCR:
- likely LOW_TEXT -> set quality_flag=LOW_TEXT and show an actionable message to provide an OCR/text version.

DOCX / PPTX:
- Best-effort support allowed, but not required for POC acceptance tests.

## Indexing quality metrics
Compute per document:
- extracted_chars
- chunks_count
- quality_flag: OK | LOW_TEXT | ERROR

Threshold suggestion:
- extracted_chars < 1500 => LOW_TEXT

## Chunking profiles
Course docs (tag=cours):
- chunk_size_tokens ~ 1200
- overlap_tokens ~ 150
- keep headings if possible

Procedures/safety (tags: procedure/checklist/securite/qualite):
- chunk_size_tokens ~ 350
- overlap_tokens ~ 50
- preserve numbered steps, warnings, control points

## Embeddings
Model (local CPU):
- sentence-transformers/static-similarity-mrl-multilingual-v1
- truncate_dim: 512 default (configurable 256/512)

Storage:
- document_chunk.embedding (pgvector) + metadata (page/slide, tag, title)

## Retrieval policy (question-driven)
Inputs:
- group_id
- query_text
- optional risk flags (keywords)
- optional missing referential item ids

Steps:
1) Determine preferred tags:
- if risk keywords or compliance/safety intent -> prefer procedure/checklist/securite/qualite
- else -> prefer cours

2) Vector search:
- top_k = 20
- filter to:
  - organisation_id
  - docs ACTIVE for group_id

3) Lexical rerank (light hybrid):
- rerank top 20 by keyword overlap with query (+ heading/step matches)

4) Injection:
- inject top 1–3 chunks
- instruct Hugo: use chunks only to craft a concrete follow-up question, not a full course

Logging:
- store rag_citation entries for each chunk used (doc_id, chunk_id, score, meta)

## Anti-absurd acceptance test
- 2 procedure docs tagged procedure/securite
- 10 procedure-focused queries
- Pass if >= 8/10 have a useful procedure/sécurité chunk in top-5
