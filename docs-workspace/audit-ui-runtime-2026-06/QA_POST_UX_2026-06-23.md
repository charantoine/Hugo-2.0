### Note — suffixe `mk1` (et `_mk2`, etc.)

**Sans objet côté UX** : dans les audits et specs de ce lot, `mk1` a parfois été cité comme s’il s’agissait d’une **notion produit** (« profil mk1 », badge « Profil mk1 », encart avec nom mk1 sur les hubs posture). C’était une **surinterprétation** de noms de fixtures de démo (`profil_conversationnel_mk1`, `diagnostic_mk1`, …).

**Réel confirmé (alignement produit) :** `mk1` / `mk2` = **marqueurs de version** ou de première itération dans le **nommage interne** des prompts et profils (convention auteur, pas concept admin). L’UI admin doit rester sur des notions compréhensibles : **nom du profil global**, **complétude n/7**, **posture**, **référentiel**, **état fallback** — jamais sur les suffixes techniques.

**Implémentation actuelle (conforme) :** le badge détail groupe utilise le **nom métier** du profil (`profile.name` via API), pas le suffixe `mk1`. Les hubs posture ont un encart générique vers les profils globaux, sans affichage de suffixe de version. **Aucun lot ultérieur** n’est requis pour « afficher mk1 » sur les hubs.

## Conclusion

Les changements sont **peu intrusifs** : présentation, hiérarchie et navigation uniquement ; logique métier et résolution backend inchangées. Alignés avec la posture **backend-first** (badges et complétude lisent les données API existantes).
